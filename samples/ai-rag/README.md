# AI / RAG with vector search

> **Version:** June 18, 2026
> **Status:** Private Preview: scaffold (code in progress)
> **Priority:** 1 (lead value prop)

## What this sample shows

A working RAG (retrieval-augmented generation) application against the Azure SQL Database container as the vector store. The container's AI-native capabilities (vector data types, vector search, embeddings) replace a separate vector database. The same application deploys against Azure SQL Database in the cloud with embeddings generated by Azure OpenAI in production.

## What you need

- The Azure SQL Database container running locally (see [Getting Started](../../docs/getting-started.md)).
- Python 3.11 or later.
- An embeddings provider for local development: [Ollama](https://ollama.com/) or [Foundry Local](https://learn.microsoft.com/en-us/azure/ai-foundry/foundry-local/). The sample defaults to Ollama running `nomic-embed-text`.

## Two legs, one schema

### Leg 1: local

```bash
# Container running on localhost:1433
# Ollama running on localhost:11434 with nomic-embed-text pulled
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
python scripts/ingest.py data/corpus.json
python -m app.query "your question here"
```

The ingest script:

1. Loads a 20-50 doc corpus.
2. Generates embeddings via Ollama.
3. Stores text and vector columns in the container using vector data types.
4. Demonstrates similarity search using `VECTOR_DISTANCE`.

### Leg 2: cloud (swap embeddings to Azure OpenAI)

```bash
# Provision Azure SQL Database and Azure OpenAI via the Azure skills collection
cp .env.cloud.example .env
python scripts/ingest.py data/corpus.json
python -m app.query "your question here"
```

The only change between the two legs is the embeddings provider configuration. The schema, the query code, and the vector search semantics are identical.

See [Azure skills collection](../azure-skills/) for the deployment flow.
